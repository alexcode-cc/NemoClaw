# 測試架構

## 測試框架

- **框架：** Vitest 4.1.0
- **覆蓋率：** V8 provider
- **報告格式：** text + json-summary

## Vitest 設定（vitest.config.ts）

兩個測試專案：

| 專案 | 包含路徑 | 語言 | 範圍 |
|------|----------|------|------|
| `cli` | `test/**/*.test.js` | JavaScript | CLI 入口點整合測試 |
| `plugin` | `nemoclaw/src/**/*.test.ts` | TypeScript | 插件單元測試 |

覆蓋率設定僅涵蓋 `nemoclaw/src/**/*.ts`，排除測試檔案。

## 執行指令

```bash
# 所有測試
npm test

# 指定專案
npx vitest run --project cli
npx vitest run --project plugin

# 單一測試檔
npx vitest run --project cli test/cli.test.js
npx vitest run --project plugin nemoclaw/src/commands/slash.test.ts

# 覆蓋率
npx vitest run --coverage

# 覆蓋率閾值檢查（CI 使用）
bash scripts/check-coverage-ratchet.sh
```

## Mock 模式

### 檔案系統 Mock

插件測試使用 in-memory Map 模擬 fs：

```typescript
// migration-state.test.ts
type FsEntry = { type: "file" | "dir" | "symlink"; content?: string; target?: string };
const store = new Map<string, FsEntry>();

// 輔助函式
addDir(path: string): void
addFile(path: string, content: string): void
addSymlink(path: string, target: string): void

// Mock 的 fs 函式
vi.mock("fs", () => ({
  existsSync: (p) => store.has(p),
  mkdirSync: (p) => store.set(p, { type: "dir" }),
  readFileSync: (p) => store.get(p)?.content,
  writeFileSync: (p, c) => store.set(p, { type: "file", content: c }),
  // ...
}));
```

### 程序執行 Mock

```javascript
// runner.test.js
const calls = [];
childProcess.spawnSync = (...args) => {
  calls.push(args);
  return { status: 0 };
};
```

### Docker Socket 偵測 Mock

```bash
# platform.test.js, runtime-shell.test.js
NEMOCLAW_TEST_SOCKET_PATHS="socket1:socket2:socket3"
```

### NPM Stub

```javascript
// install-preflight.test.js
writeNpmStub(fakeBin, installSnippet)
// 處理 --version, config get prefix, install/link
```

### 測試隔離模式

| 技術 | 用途 |
|------|------|
| HOME 覆蓋 | `/tmp/nemoclaw-*-${Date.now()}` |
| PATH 限制 | `TEST_SYSTEM_PATH = "/usr/bin:/bin"` |
| 暫存清理 | `fs.rmSync(tmpDir, { recursive: true, force: true })` |
| require cache 清除 | `delete require.cache[modulePath]` |
| ANSI 剝離 | `sed 's/\x1b\[[0-9;]*m//g'` |

沙箱驗證測試：
- 插件安裝確認
- Blueprint YAML 驗證
- 快照建立/還原
- 遷移清單驗證

### test/Dockerfile.sandbox

測試用沙箱映像，與正式 Dockerfile 的差異：
- 修補 OpenClaw 模型目錄（替換為 Nemotron 模型）
- 設定 reasoning=true、maxTokens=8192
- 建立遷移測試用的固定設定：
  - 外部狀態目錄（`OPENCLAW_STATE_DIR` 測試）
  - 外部設定檔（`OPENCLAW_CONFIG_PATH` 測試）
  - Symlinks 和 Hooks（tar 保留驗證）

## CI/CD 流程

### GitHub Actions 工作流程

#### pr.yaml（Pull Request）

三個平行 Job：

| Job | 時限 | 動作 |
|-----|------|------|
| `lint` | 5 分鐘 | `npx prek run --all-files --stage pre-push`（涵蓋所有鉤子：格式化、lint、型別檢查）；含 hadolint 二進位安裝（SHA256 驗證） |
| `test-unit` | 10 分鐘 | `npm test` + 覆蓋率閾值 |
| `test-e2e-sandbox` | 15 分鐘 | Docker build test/Dockerfile.sandbox + e2e-test.sh |

所有 actions 已升級至 v6（checkout、setup-node、setup-python、setup-uv）。

#### commit-lint.yaml

- 觸發：PR events + title 編輯
- 動作：commitlint 驗證 commit 範圍；Squash merge 時驗證 PR 標題

#### nightly-e2e.yaml

- 觸發：每日 UTC 00:00 或手動
- 需要 Secret：`NVIDIA_API_KEY`
- 使用非互動模式（`NEMOCLAW_NON_INTERACTIVE=1`）
- 失敗時上傳安裝日誌

#### docs-preview-pr.yaml + docs-preview-deploy.yaml（拆分架構）

原本的 `docs.yaml` 已移除（冗餘），文件工作流程拆分為兩步：

- **docs-preview-pr.yaml**（PR 觸發）：嚴格建置文件（`sphinx-build -W`）、上傳 HTML artifact（3 天保留）、儲存 PR metadata
- **docs-preview-deploy.yaml**（透過 `workflow_run` 觸發）：具備 gh-pages 寫入權限、過濾 fork PR（`same-repo != 'true'` 時跳過部署）、使用 `rossjrw/pr-preview-action` 部署預覽

#### docker-pin-check.yaml

- 觸發：每週一 UTC 09:00
- 動作：比對 Dockerfile 釘選摘要與 Docker Hub 最新 manifest，過期時失敗

#### pr-limit.yaml

PR 數量速率限制

## 覆蓋率管理

使用 `scripts/check-coverage-ratchet.sh` 強制執行最低覆蓋率門檻。CI 中每次 PR 都會檢查覆蓋率是否低於門檻。
