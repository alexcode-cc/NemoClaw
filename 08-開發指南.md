# 開發指南

## 環境需求

| 工具 | 版本 | 用途 |
|------|------|------|
| Node.js | ≥ 22 | CLI 與 TypeScript 插件 |
| npm | ≥ 10 | 套件管理 |
| Python | ≥ 3.11 | Blueprint 與文件建置 |
| Docker | 最新穩定版 | 沙箱容器運行 |
| [uv](https://docs.astral.sh/uv/) | 最新 | Python 相依管理 |
| hadolint | 最新 | Dockerfile linter（macOS：`brew install hadolint`） |

## 初始設定

```bash
# 1. 安裝根層級相依（包含 prek 鉤子安裝）
npm install

# 2. 安裝並建置 TypeScript 插件
cd nemoclaw && npm install && npm run build && cd ..

# 3. 安裝 Python 相依
cd nemoclaw-blueprint && uv sync && cd ..
```

## Git Hooks 詳解（prek）

所有 Git Hooks 由 [prek](https://prek.j178.dev/) 管理，`npm install` 時自動安裝。

### 鉤子階段與優先序

同一優先序的鉤子會**平行執行**。

#### pre-commit

| 優先序 | 鉤子 | 說明 |
|--------|------|------|
| 0 | trailing-whitespace, end-of-file-fixer, mixed-line-ending (LF), no-force-added-ignored | 檔案修正 |
| 5 | shfmt (2-space, -ci, -bn), ruff-format, prettier | 格式化 |
| 6 | eslint --fix, ruff --fix | 自動修復 |
| 10 | shellcheck, hadolint, markdownlint-cli2, check-merge-conflict, check-large-files, check-yaml, check-toml, check-json, detect-private-key, gitleaks, spdx-headers | 靜態分析與安全掃描 |
| 20 | vitest run --project plugin | 插件單元測試 |

#### commit-msg

commitlint — 強制 Conventional Commits 格式

#### pre-push

- `tsc --noEmit`（nemoclaw/）— TypeScript 型別檢查
- `pyright`（nemoclaw-blueprint/）— Python 型別檢查

### 新增 Lint 設定檔

| 檔案 | 用途 |
|------|------|
| `.markdownlint-cli2.yaml` | Markdown lint 規則：停用 MD004（ul-style）、MD013（line-length）、MD014（commands-show-output）、MD024（no-duplicate-heading）、MD033（no-inline-html）、MD060（table-column-style） |
| `.shellcheckrc` | Shell lint 設定：停用 SC2059（ANSI 色碼變數在 printf 中）、SC1091（動態 source 路徑） |

### Husky 遷移注意事項

若從舊版 Husky 設定遷移，Git 可能忽略 `.git/hooks`。執行：

```bash
git config --unset core.hooksPath
npm install    # prek install 會重新註冊鉤子
```

### 排除路徑

```
^(nemoclaw/dist/|nemoclaw/node_modules/|docs/_build/|\.venv/|uv\.lock$)
```

## 提交訊息規範

遵循 AngularJS Git Commit Message Conventions，所有提交訊息使用**繁體中文**。

### 格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

### type（必填）

| 類型 | 說明 |
|------|------|
| `feat` | 新功能 |
| `fix` | 修復錯誤 |
| `docs` | 文件變更 |
| `style` | 格式調整（不影響程式邏輯） |
| `refactor` | 重構（非新功能、非修復） |
| `test` | 新增或修改測試 |
| `chore` | 建置流程或輔助工具變更 |
| `perf` | 效能改善 |
| `ci` | CI/CD 變更 |

### scope（選填）

影響範圍：`cli`、`blueprint`、`plugin`、`docs`

### 範例

```
feat(cli): 新增 --profile 旗標至 onboard 指令

支援使用者在 onboard 階段直接指定推論設定檔，
省去後續手動設定的步驟。

BREAKING CHANGE: onboard 指令的預設行為已變更，
不再自動選擇 default 設定檔。
```

```
fix(blueprint): 修正缺少 API 金鑰時的錯誤處理
```

## 文件撰寫規範

### 格式

- 使用 MyST Markdown（Sphinx 相容的 CommonMark 超集）
- 每頁以 YAML frontmatter 開頭（title, description, content type）
- 每個句子獨立一行（使 diff 可讀）

### 風格

- 主動語態：「CLI 建立閘道」而非「閘道被 CLI 建立」
- 第二人稱「你」
- 現在式
- 不使用修飾詞：「simply」、「just」、「easily」
- 不使用 emoji

### 專有名詞大小寫

| 正確 | 錯誤 |
|------|------|
| NemoClaw | nemoclaw（在散文中）、Nemoclaw |
| OpenClaw | openclaw（在散文中）、Openclaw |
| OpenShell | Open Shell、openshell |
| NVIDIA | Nvidia、nvidia |
| CLI | cli、Cli |
| API key | api key、API Key |
| YAML | yaml、Yaml |

### 文件更新流程

1. 編輯 `docs/` 目錄下的頁面
2. **永遠不要**直接編輯 `.agents/skills/docs/`（自動產生）
3. 變更後重新產生 Skills：
   ```bash
   python scripts/docs-to-skills.py docs/ .agents/skills/docs/ --prefix nemoclaw
   ```
4. 本地建置驗證：`make docs`

## Pull Request 流程

1. 從 `main` 建立功能分支
2. 進行變更並撰寫測試
3. 執行 `make check` 和 `npm test` 驗證
4. 開啟 PR

### PR 檢查清單

- [ ] 遵循 Contributing 指南
- [ ] 程式碼已格式化
- [ ] 測試已通過
- [ ] 不包含敏感資訊
- [ ] 若變更影響使用者行為，已更新文件

### 重要限制

- **不加外部連結**：不要在文件或程式碼中加入第三方程式庫、社群合集或非官方資源的連結
- **PR 數量限制**：同時開啟的 PR 不超過 10 個
- **安全漏洞**：遵循 SECURITY.md，不要使用 GitHub Issues
